Broken Access Control Security Project

This project demonstrates how Broken Access Control vulnerabilities can be tested in web applications. The project explains how attackers exploit access control weaknesses such as IDOR, Horizontal Privilege Escalation, Vertical Privilege Escalation, and JWT Manipulation.

1. Lab Setup

Tools required:

  • Burp Suite
  • Google Chrome/ Firefox Browser
  • Test Web Application

Configure proxy:

Proxy Address: 127.0.0.1
Port: 8080

All traffic from the browser will now pass through Burp Suite, allowing us to intercept and modify requests.

2. IDOR (Insecure Direct Object Reference)

IDOR occurs when an application exposes internal object IDs without proper authorization checks. Attackers can modify these IDs to access other users' data.

Example API endpoint:

GET /api/orders/1

If an attacker changes the request:

GET /api/orders/2
GET /api/orders/3

If the server returns other users' order details, then the application is vulnerable to IDOR.

Testing Steps

  1. Login to the application
  2. Capture request using Burp Suite
  3. Send request to Repeater
  4. Modify the order ID
  5. Check if other users' data is returned

3. Horizontal Privilege Escalation

Horizontal privilege escalation occurs when one user can access another user's data with the same privilege level.

Example request:

GET /api/user/101/profile

Attacker changes request to:

GET /api/user/102/profile

If the application returns user 102 data, the vulnerability exists.

4. Vertical Privilege Escalation

Vertical privilege escalation occurs when a normal user gains access to admin-level functionality.

Example admin endpoint:

GET /api/admin/users

If a normal user can access this endpoint and retrieve admin data, it means the application has a vertical privilege escalation vulnerability.

Testing Steps

  1. Login as a normal user
  2. Capture a request in Burp Suite
  3. Modify the endpoint to admin API
  4. Send the request
  5. If data is returned, the vulnerability exists

5. JWT Token Manipulation

Many applications use JWT tokens for authentication. A JWT consists of three parts: 

HEADER.PAYLOAD.SIGNATURE

Example payload:

{
"user": "alice",
"role": "user"
}

An attacker may modify it to:

{
"user": "alice",
"role": "admin"
}

If the server does not properly validate the signature, the attacker may gain admin privileges.

Testing Steps

  1. Intercept request containing JWT token
  2. Decode token
  3. Modify payload
  4. Resend request
  5. Check if admin access is granted

6. Security Impact

  • Unauthorized data access
  • Data leakage
  • Account takeover
  • Admin privilege abuse
  • Financial loss

7. Prevention Methods

  • Implement proper server-side authorization checks
  • Use indirect object references
  • Validate JWT tokens securely
  • Use role-based access control
  • Log and monitor suspicious activity

8. Conclusion

Broken Access Control is one of the most critical vulnerabilities in modern web applications. Through this project, we demonstrated how attackers exploit access control weaknesses using different techniques. Proper validation and authorization mechanisms must be implemented to prevent such vulnerabilities.
